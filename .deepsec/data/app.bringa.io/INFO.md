# app.bringa.io

## What this codebase does

Open-source community sharing and borrowing portal for clubs, associations, and local operators. It is a static-export Next.js/React app backed by Supabase Auth, Postgres, Storage, RLS, and two Deno Edge Functions for Telegram notifications. Fork operators configure public deployment identity, legal text, Supabase URL, and publishable key through generated app config; secrets belong only in local env files, deployment secrets, or Supabase function secrets.

## Auth shape

- `supabase` in `src/lib/supabaseclient.ts` is the browser client. It uses public generated config and switches to `createLocalDemoSupabaseClient` only when `isLocalDemoModeEnabled` sees `NODE_ENV === "development"`.
- `ProtectedRoute` and `decideProtectedRoute` are UI gates: a session must exist, the profile must be `profile_valid`, and display name fields must be complete before normal pages render.
- `useIsAdmin` calls the `is_admin` RPC, and `buildAdminRouteGate` only gates admin UI. Database RPCs and RLS remain the authority for admin access.
- Supabase SQL helpers `is_validated`, `is_admin`, `verify_and_apply_invite`, and `prevent_profile_escalation` are core trust primitives. `profile_valid` and `invited_by_code` should only change through trusted RPC flow.
- Mutations should go through SECURITY DEFINER RPCs such as `create_item`, `update_item`, `borrow_item`, `return_item`, `promote_admin`, `demote_admin`, `set_profile_validation`, moderation review/apply RPCs, and account deletion RPCs.

## Threat model

Highest-impact attacks are becoming admin, validating an uninvited profile, reading or changing other users' private profile/admin/moderation/deletion data, or abusing trusted service keys in Edge Functions. A normal validated user should not be able to bypass RPC ownership checks for item edits, borrow/return state, moderation queues, visibility, data export, or account deletion. Storage abuse matters because item images are public once accepted; upload paths, MIME/type limits, and item image metadata must stay bound to the authenticated uploader.

## Project-specific patterns to flag

- Client-side admin gating is not sufficient. Flag admin page code that directly mutates `profiles`, `admins`, `items`, moderation tables, deletion requests, backup runs, or notification tables instead of calling an RPC with server-side `public.is_admin()` checks.
- SECURITY DEFINER functions granted to `authenticated` must explicitly check `auth.uid()` plus `public.is_validated()` or `public.is_admin()` as appropriate, and must not trust profile/admin values provided by the client.
- `profile_valid`, `invited_by_code`, and admin membership are sensitive. Flag any direct browser updates, broad RLS policies, or trigger changes that let users self-validate, view invite codes, promote themselves, or revoke safeguards against self-lockout.
- Item image uploads must stay in bucket `items`, under `<auth.uid()>/<image-id>/detail.webp` and `thumb.webp`, with matching `item_images` rows created only by RPCs. Flag arbitrary bucket/path uploads, direct `item_images` writes, or accepting client-supplied public URLs without path validation.
- Edge Functions may use `SUPABASE_SECRET_KEY`, `SUPABASE_SECRET_KEYS`, legacy `SUPABASE_SERVICE_ROLE_KEY`, Telegram bot tokens, and chat IDs only from Deno env. Flag hardcoded secrets, logging secret values, unauthenticated status-write paths, or service-key behavior moved into browser code.

## Known false-positives

- `src/lib/local-demo-supabase.ts` intentionally contains fake users, fake tokens, permissive in-memory RPC stubs, and data URLs for local browser demos. It is guarded by `isLocalDemoModeEnabled` and must remain development-only.
- `src/config/bringa.config.generated.json` intentionally contains a Supabase URL and publishable key; these are public browser values, not service-role secrets.
- The `items` Storage bucket and item image URLs are intentionally public for displayed item photos; write/delete authority belongs in Storage policies and RPC metadata checks.
- Supabase Edge Functions read secret env vars and call Telegram. Env var names and generic error logs are expected; literal token values are not.
- `.deepsec/data/app.bringa.io/` contains scanner state and reports generated by DeepSec, not application runtime code.
